## 개요

본 보고서는 울산광역시 시각장애인복지관의 구형 웹사이트와 신형 웹사이트를 시각장애인 접근성 관점에서 비교 분석한 문서입니다. 웹 접근성 표준(WCAG 2.1, 한국형 웹 콘텐츠 접근성 지침 2.1)을 기준으로 평가하였습니다.

## 구형 사이트 분석 (울산광역시시각장애인복지관1.html)

### 장점

#### 1. 접근성 측면
- **본문 바로가기 링크 제공**: `<a id="skip-nav">` 를 통해 키보드 사용자가 반복적인 메뉴를 건너뛸 수 있음
- **스크린 리더 전용 텍스트**: `.sr-only` 클래스를 사용하여 화면에는 보이지 않지만 스크린 리더로는 읽을 수 있는 텍스트 제공
- **의미있는 대체 텍스트**: 이미지에 적절한 alt 속성 제공 (예: `alt="울산광역시시각장애인복지관 유튜브채널 바로가기-새창"`)
- **명확한 언어 설정**: `<html lang="ko">` 로 페이지 언어를 명시
- **ARIA 속성 사용**: `aria-hidden="true"` 를 사용하여 장식용 아이콘을 스크린 리더에서 숨김

#### 2. 네비게이션 구조
- **계층적 메뉴 구조**: 7개의 주요 메뉴와 각각의 하위 메뉴로 체계적으로 구성
- **일관된 메뉴 구조**: `js-nav-system` 클래스를 통한 일관된 네비게이션 패턴

#### 3. 콘텐츠 구성
- **명확한 섹션 구분**: semantic HTML (`<header>`, `<nav>`, `<section>`, `<footer>`) 사용
- **정보의 체계적 배치**: 공지사항, 재활정보, 문의하기, 포토갤러리 등 섹션별 명확한 구분

### 단점

#### 1. 접근성 문제
- **메뉴 구조의 복잡성**: 하위 메뉴가 숨겨진 상태에서 시작하여 마우스 호버 또는 클릭이 필요할 수 있음
- **캐러셀/슬라이더 제어**: 팝업존 롤링에 정지/재생 버튼은 있으나, 키보드로 개별 슬라이드 탐색이 어려울 수 있음
- **폼 요소 레이블 부족**: 관련기관, 복지재단 선택 드롭다운에 `<label>` 은 있으나 시각적으로만 표현됨

#### 2. 보안 문제
- **외부 리소스 의존**: 구글 사이트 인증(`meta name="google-site-verification"`)은 있으나, 외부 스크립트 및 리소스에 대한 무결성 검증 부재
- **HTTP 링크 혼용**: 일부 외부 링크가 HTTP로 되어 있어 보안 위험 (예: `http://www.ubu.or.kr/`)
- **인라인 JavaScript**: `onclick` 속성 사용으로 CSP(Content Security Policy) 적용 시 문제 발생 가능

#### 3. 사용성 문제
- **텍스트 크기 고정**: viewport가 고정 폭(width=1100)으로 설정되어 모바일 반응형이 제한적
- **날짜 형식 불일치**: 게시글 날짜가 "11/11", "11/07" 등 축약형으로 표시되어 연도 정보 부재
- **이미지 기반 버튼**: "더보기" 버튼이 이미지로 구현되어 확대 시 품질 저하

## 신형 사이트 분석 (울산광역시시각장애인복지관.html)

### 장점

#### 1. 현대적 프레임워크
- **CodeIgniter 4 기반**: 최신 PHP 프레임워크(v4.6.0) 사용으로 보안성 향상
- **개발 도구 통합**: Debug Toolbar를 통한 개발 및 디버깅 용이

#### 2. 데이터베이스 기반 콘텐츠
- **동적 콘텐츠 관리**: 정적 HTML이 아닌 데이터베이스 기반으로 콘텐츠 관리 가능
- **파일 관리 시스템**: 파일 업로드 및 대체 텍스트 관리 구조 포함

### 단점

#### 1. 심각한 접근성 문제
- **개발자 도구 노출**: Debug Toolbar가 프로덕션 환경에서도 활성화되어 있음 (7000줄 이상의 디버그 코드)
- **기본 접근성 기능 부재**: 본문 분석 결과 구형 사이트에 있던 접근성 기능들이 보이지 않음
  - 본문 바로가기 링크 미확인
  - 스크린 리더 전용 텍스트 미확인
  - Semantic HTML 구조 미확인

#### 2. 보안 문제
- **개발 모드 노출**: `Environment: development` 상태로 배포
- **민감 정보 노출**: 데이터베이스 쿼리, 세션 데이터, 시스템 구성 정보 등이 클라이언트에 노출
- **보안 헤더 부재**: Content Security Policy 미활성화
- **디버그 정보 노출**: 파일 경로, 서버 구조, 라이브러리 버전 등 상세 정보 노출

#### 3. 성능 문제
- **과도한 페이지 크기**: 디버그 툴바로 인해 페이지 크기가 불필요하게 증가 (약 7000+ 줄)
- **불필요한 JavaScript**: 디버그 관련 JavaScript 코드가 대량 포함

## 전문가 관점 평가

### 웹 접근성 전문가 관점

#### 구형 사이트
**강점:**
1. **WCAG 2.1 Level A 준수 노력**: 기본적인 접근성 요구사항을 충족하기 위한 노력이 보임
2. **스크린 리더 최적화**: `.sr-only` 클래스와 적절한 ARIA 속성 사용
3. **키보드 네비게이션**: 본문 바로가기 기능으로 키보드 사용자 편의 제공
4. **구조화된 콘텐츠**: Semantic HTML5 태그 사용

**약점:**
1. **동적 콘텐츠 접근성**: 캐러셀과 드롭다운 메뉴의 키보드 접근성 개선 필요
2. **포커스 관리**: 시각적 포커스 표시가 명확하지 않을 수 있음
3. **색상 대비**: 색상 대비율 검증 필요 (코드만으로 확인 불가)
4. **에러 처리**: 폼 유효성 검사 시 스크린 리더 친화적 에러 메시지 필요

#### 신형 사이트
**강점:**
1. **확장 가능한 구조**: CMS 기반으로 접근성 기능 일관성 있게 적용 가능
2. **대체 텍스트 관리**: DB 레벨에서 alt_text 관리

**약점:**
1. **프로덕션 준비 미완료**: 개발 환경 그대로 노출
2. **접근성 구현 미확인**: 기본 접근성 기능 구현 여부 불명확
3. **성능 이슈**: 디버그 코드로 인한 페이지 로딩 지연 가능

### 보안 전문가 관점

#### 구형 사이트
**강점:**
1. **최소 공개 원칙**: 불필요한 시스템 정보 노출 최소화
2. **프로덕션 환경**: 정식 서비스 환경으로 배포

**약점:**
1. **HTTP 링크**: 외부 링크에 HTTP 사용으로 중간자 공격 위험
2. **인라인 스크립트**: CSP 적용 어려움
3. **외부 리소스 검증**: Subresource Integrity (SRI) 미사용
4. **오래된 라이브러리**: jQuery 등 라이브러리 버전 확인 필요

#### 신형 사이트
**강점:**
1. **현대적 프레임워크**: 최신 보안 패치가 적용된 CodeIgniter 4 사용
2. **구조화된 보안**: 프레임워크 레벨의 보안 기능 활용 가능

**치명적 약점:**
1. **개발 모드 노출**: 실제 서비스에서 절대 발생해서는 안 되는 심각한 보안 결함
2. **시스템 정보 유출**: 
   - 데이터베이스 쿼리 노출
   - 파일 시스템 구조 노출
   - 라이브러리 버전 정보 노출
   - 세션 관리 정보 노출
3. **공격 벡터 제공**: 디버그 정보가 공격자에게 시스템 취약점 파악의 단서 제공

## 일반인 관점 평가

### 사용 편의성

#### 구형 사이트
**좋은 점:**
1. **직관적 메뉴**: 7개 주요 메뉴가 명확하게 구분되어 있음
   - 복지관안내, 프로그램안내, 시각장애인의 이해, 음성정보서비스, 후원/자원봉사, 복지자료실, 게시판
2. **명확한 정보 계층**: 각 메뉴의 하위 항목이 논리적으로 구성
3. **주요 기능 빠른 접근**: 프로그램 시간표, 점심메뉴, 행사일정표 등 자주 사용하는 기능을 메인에 배치
4. **소셜 미디어 연결**: YouTube 채널로 바로 이동 가능

**개선 필요:**
1. **모바일 최적화 부족**: 고정폭 레이아웃으로 스마트폰에서 사용 불편
2. **날짜 표시**: 연도가 없어 오래된 게시물 구분 어려움
3. **검색 기능 부재**: 사이트 내 검색 기능이 보이지 않음

#### 신형 사이트
**좋은 점:**
1. **콘텐츠 관리 용이성**: 관리자가 콘텐츠를 쉽게 수정 가능한 구조

**개선 필요:**
1. **디버그 정보 방해**: 일반 사용자에게 불필요한 개발자 도구가 화면을 차지
2. **성능 저하**: 불필요한 코드로 인한 로딩 속도 저하
3. **완성도**: 아직 개발 단계로 보임

### 시각장애인 사용자 관점

#### 구형 사이트
**좋은 점:**
1. **스크린 리더 친화적**: 
   - 본문 바로가기로 반복 콘텐츠 스킵 가능
   - 이미지에 적절한 설명 제공
   - 링크 목적이 명확함 ("새창", "더보기" 등 명시)
2. **논리적 탐색 순서**: Tab 키로 순차적 탐색 가능
3. **음성정보서비스**: 복지관 특성에 맞는 음성 서비스 메뉴 제공

**개선 필요:**
1. **드롭다운 메뉴**: 하위 메뉴 접근 시 마우스 호버 필요 가능성
2. **캐러셀 제어**: 팝업존 자동 롤링이 스크린 리더 사용에 방해될 수 있음
3. **폼 입력 안내**: 입력 필드의 필수/선택 여부, 형식 안내 부족

#### 신형 사이트
**우려사항:**
1. **접근성 검증 불가**: 실제 렌더링된 HTML 구조를 확인할 수 없어 평가 어려움
2. **디버그 노이즈**: 스크린 리더가 디버그 정보를 읽을 가능성

## 보완 권장사항

### 즉시 조치 필요 (Critical)

#### 신형 사이트
1. **개발 모드 비활성화** (최우선)
   ```php
   // .env 파일 또는 환경 설정
   CI_ENVIRONMENT = production
   ```

2. **Debug Toolbar 제거**
   ```php
   // app/Config/Toolbar.php
   public array $collectors = []; // 프로덕션에서 빈 배열
   ```

3. **보안 헤더 설정**
   - Content Security Policy 활성화
   - X-Frame-Options 설정
   - X-Content-Type-Options 설정

### 단기 개선 (High Priority)

#### 신형 사이트
1. **접근성 기능 구현**
   - 본문 바로가기 링크 추가
   - 스크린 리더 전용 텍스트 클래스 구현
   - Semantic HTML 구조 적용
   - ARIA 속성 적절히 사용

2. **성능 최적화**
   - 불필요한 디버그 코드 제거
   - CSS/JS 파일 최소화 및 결합
   - 이미지 최적화

#### 구형 사이트
1. **보안 강화**
   - 모든 외부 링크를 HTTPS로 변경
   - SRI (Subresource Integrity) 적용
   - CSP 헤더 추가

2. **반응형 디자인**
   ```html
   <meta name="viewport" content="width=device-width, initial-scale=1.0">
   ```

### 중기 개선 (Medium Priority)

#### 공통
1. **메뉴 접근성 개선**
   - 키보드로 모든 하위 메뉴 접근 가능하도록 개선
   - 현재 위치 표시 (breadcrumb 또는 active 상태)
   - 메뉴 구조를 WAI-ARIA Authoring Practices 가이드에 맞게 구현

2. **폼 접근성**
   ```html
   <label for="search-input">검색어 입력</label>
   <input id="search-input" type="text" aria-required="true" 
          aria-describedby="search-help">
   <div id="search-help" class="sr-only">
     검색어를 입력하고 엔터를 누르세요
   </div>
   ```

3. **에러 메시지**
   ```html
   <div role="alert" aria-live="assertive">
     필수 항목을 입력해주세요
   </div>
   ```

4. **캐러셀 개선**
   - 자동 재생 기본값을 '정지'로 설정
   - 각 슬라이드로 직접 이동할 수 있는 컨트롤 제공
   - 남은 시간 표시 (선택사항)

### 장기 개선 (Low Priority)

1. **다국어 지원**

2. **고대비 모드**

3. **글자 크기 조절**

   - 복잡한 표나 그래프에 대한 대체 설명

5. **사용자 맞춤 설정**

   - 간편한 모드/상세한 모드 전환

## 결론

**구형 사이트**는 기본적인 웹 접근성 표준을 따르려는 노력이 돋보이며, 시각장애인 사용자를 위한 기능들이 구현되어 있습니다. 그러나 보안 및 모바일 대응에 개선이 필요합니다.

**신형 사이트**는 현대적인 프레임워크 기반으로 확장성과 유지보수성이 우수하나, 현재 상태로는 **프로덕션 환경에 배포해서는 안 됩니다**. 개발 모드 노출은 심각한 보안 위험이며, 접근성 기능 구현도 확인되지 않았습니다.

**권장사항**: 
1. 신형 사이트는 즉시 개발 모드를 비활성화하고, 구형 사이트의 모든 접근성 기능을 이식한 후 배포
2. 두 사이트 모두 HTTPS 전면 적용, CSP 설정 등 기본 보안 조치 필요
3. 정기적인 접근성 감사 및 실제 사용자 테스트 시행
4. 웹 접근성 인증마크(WA) 획득 목표 설정 권장

이 보고서가 울산광역시 시각장애인복지관의 웹사이트 개선에 도움이 되기를 바랍니다.
